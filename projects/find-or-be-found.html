Find or be Found

Overview

Find or Be Found is my debut premium game, both as a game developer and as a co-founder of Studio Downstairs. As the sole programmer, I led the planning and execution of the large-scale project, which proved both challenging and highly rewarding. I gained hands-on experience across the full development cycle, from prototyping to release and post-launch support. Since the project was self-funded, my availability varied. Throughout development, I contributed between 30% and 100% of my time to the project. During the finalsix months, we collaborated with two interns in design & 3D art, an experience that was valuable for both them and me.

Networking

This was my first time diving into networking development. I started by exploring different solutions that would best fit the project and ultimately decided on FishNet Networking Evolved. I chose FishNet because it felt flexible and easy to work with. It also has an active Discord community where you can get help with specific questions, which was a lifesaver for me since I used it A LOT.

We decided to make Find or be Found multiplayer partway through development, after already building a single-player prototype. This meant I had to introduce networking into an existing structure, and since I didn't yet fully understand how to think about or prioritize systems in a networked game, it led to a lot of “silver tape” and spaghetti solutions early on. That, in turn, resulted in plenty of debugging and refactoring along the way.

We don't use prediction in the game since the two players never directly interact with each other, which simplifies the implementation. Even so, I still needed to build a lobby system that integrates with Steam's API, as well as implement a voice chat solution with support for muting and unmuting players. There was definitely a steep learning curve in understanding RPCs and how to properly structure networked systems. After a lot of desync issues and objects not behaving as expected, I eventually managed to get everything working well enough.

Item Generation

A large part of the game revolves around generating the items players can discover, so I needed a reliable way to ensure each one was both valid and unique. I also needed a system to verify whether a player had found the correct item. To solve this, I designed an ID-based system where every item combination is assigned a unique identifier. This allows me to efficiently store and retrieve item data, track how many instances of a specific item have spawned, and measure how similar different items are to one another.

For example, if we take a bottle, we can see that the type is "bottle," which corresponds to number #6. The difficulty in this case is #1. The base object is the whole bottle body, numbered #3. The first material is the green base material (#4), and the supporting material is the text label (#2). On top of the bottle is a lid, which is the hook object (#1). Lastly, there's the symbol (#5), which gives the object its ID: #6134215.

With this ID system, validating a delivery becomes easy: I simply compare the ID of the object the player handed in against the expected ID for the quest, and if they match, the objective is complete. Since every relevant property (type, difficulty, base object, materials, and symbol) is already baked into the ID itself, there's no need for separate checks or special-case logic per quest, one comparison covers it all.

Tools

Layout Wizard

We use a semi-random generation approach when creating levels. Rooms are hand-crafted and then connected into a layout, which is later randomized to create variation. To make the level layout pipeline more efficient, I implemented a custom tool that handles saving and loading layouts. This wizard stores all layout data in a ScriptableObject, which is assigned a name and a number. The number acts as an identifier for the layout in-game.

You can then reference this ScriptableObject to load and update a specific layout. The tool itself is a custom Unity window, accessible from the top toolbar. It provides helpful error messages to guide you if something is missing or set up incorrectly. Once everything is in place, I can randomize between ScriptableObjects and spawn a specific layout as needed.

Object visualizer

I developed an object visualization tool to streamline scene placement and debugging. You can toggle the visuals for convenience, filter by object type, and set the maximum spawn difficulty. The visualizer dynamically syncs with the target object's position, rotation, and scale.

Take Aways

This project taught me the importance of building flexible and maintainable systems. By implementing a robust ID system and custom tools, we were able to significantly reduce development time while improving the overall quality of the game's level design. It also gave me a stronger understanding of how networked systems work, as well as what I would approach differently in future projects. Going through the entire pipeline of creating a game has made me much more confident as a developer, and overall it was a very fulfilling experience.
